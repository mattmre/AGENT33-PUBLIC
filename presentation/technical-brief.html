01Overview

AGENT-33 is an async Python multi-agent orchestration framework built on top of FastAPI. The runtime accepts agent invocations and workflow runs over HTTP, WebSocket, and Server-Sent Events; routes them through a DAG executor against a registry of LLM-backed agents; and persists the resulting traces, evidence, lineage, and review state to durable stores.

The platform is structured as three concentric layers:

  1. Product surfaces — agents, workflows, skills, packs, tools, code execution. The bits operators interact with.
  2. Platform — LLM router, memory, security, autonomy, review, release, improvement, evaluation. The bits that govern the product surfaces.
  3. Foundation — observability, scaling, sessions, sandboxing, plugins, provenance, training. The bits that keep the platform honest.
Reading order: the white paper covers the design rationale; this brief covers the runtime mechanics. The architecture deck has the Mermaid diagrams.

02Runtime topology

The FastAPI application is stateless. Durable state lives in PostgreSQL (with pgvector), Redis, NATS, and the local filesystem. Optional Ollama provides local LLM inference. Every subsystem is constructed by the lifespan and stored on app.state.*; shutdown reverses the order so that the database connection pool is the last thing to close.

03Tech stack

Application
FastAPI async Python with Uvicorn / asgi-lifespan, strict mypy, Ruff linting, pytest in async mode. Build via Hatchling (engine/pyproject.toml).
Persistence
PostgreSQL 14+ with the pgvector extension. Async via SQLAlchemy 2.x. Migrations managed by Alembic (engine/alembic/versions/).
Memory
LongTermMemory writes 1536-dim vectors into pgvector with an IVF-Flat cosine index. Short-term ring buffer on top. BM25Index + reciprocal rank fusion for hybrid retrieval.
Caching
Redis 6+ async client with TTL-based rate limiters, ephemeral coordination, and per-tenant locks. Falls back to InProcessCache in lite mode.
Event bus
NATS 2.10+, JetStream-capable. Carries messaging adapter events (Telegram / Discord / Slack / WhatsApp) and sub-agent dispatch. Falls back to InProcessMessageBus in lite mode.
LLM router
Ollama-friendly model router with OpenAI-compatible adapter. The provider catalogue auto-registers 20+ providers from environment variables, including local-inference backends (Ollama, llama.cpp, LM Studio, AirLLM).
Embeddings
Pluggable provider with an LRU cache. Default path uses local Ollama embedding models; cloud providers (OpenAI, Voyage) registered through the same interface.
Frontend
React 18 + TypeScript single-page application built with Vite, served by Nginx in production. The operator console for sessions, agents, workflows, packs, traces, and release operations.
CLI
agent33 shipped as a console-script entry point. status, sessions, bench, packs, configuration introspection.
Observability
Structured structlog JSON to stdout, Prometheus metrics at /metrics, rolling MetricsCollector, rule-based AlertManager, ExecutionLineage, ExecutionReplay, and a structured TraceCollector.
Container runtime
Multi-stage Dockerfile with non-root user, minimal base. Compose stack in engine/docker-compose.yml with profiles for local-ollama, dev, agent-os, integrations, and gpu.
Orchestration
Kubernetes overlays under deploy/k8s/ (kustomize-based). Base manifests are cluster-agnostic; the production overlay adds replica counts, ingress hostnames, and TLS configuration. HPA shipped with conservative defaults.

04Lifespan order

The FastAPI lifespan in engine/src/agent33/main.py brings subsystems up in dependency order. Each step can read from previous steps. Shutdown is the reverse.

Step 1-3

Bootstrap

paths · secrets · DB

Resolve runtime state paths, validate secrets (exit on insecure JWT in prod), open the async PostgreSQL engine and create the vector extension.

Step 4-6

State stores

Redis · NATS · guards

Async Redis (fallback to in-process cache), async NATS (fallback to in-process bus), scaling guards for cron ownership.

Step 7-9

Registries

agents · tools · packs · skills

Agent registry auto-discovers JSON definitions. Tool registry scans entry points and validates schemas. Pack and skill registries scan filesystems.

Step 10-12

Execution + LLM

sandbox · router · embeddings

Code executor with adapters, model router, embedding provider with LRU cache, BM25 index, hybrid searcher, RAG pipeline, progressive recall.

Step 13-14

Governance + skill injection

approvals · sessions

Approval service, token manager, tool governance loads ~/.agent33/approved-tools.json. Capability packs, skill injector, hybrid skill matcher wired into AgentRuntime.

Step 15-16

Integrations + transports

ingestion · messaging · MCP · WS / SSE

Knowledge ingestion cron, web research, voice sidecar probe, messaging adapters, MCP service bridge, WebSocket and SSE managers, agent-to-workflow bridge.

Fallback discipline: Subsystems that fail to initialise log a warning and fall back to an in-process alternative where one exists. The application starts in lite mode if PostgreSQL, Redis, and NATS are all unavailable.

05Agent runtime

Agents are declarative JSON files auto-loaded from engine/agent-definitions/ by the AgentRegistry. Six reference agents ship in the box:

  • orchestrator — high-level director, hand-off and evidence-aware
  • director — project-level planning and decomposition
  • worker — code worker for implementation tasks
  • qa — quality assurance, evaluation, regression triage
  • researcher — research and source synthesis
  • browser-agent — web research with the headless browser tool

Tool-use loop

AgentRuntime.invoke() opens a trace, runs progressive recall against memory, builds the system prompt (skills injected via L0 / L1 / L2 disclosure), calls the LLM with the allowed tools, executes tool calls via the governed ToolRegistry.validated_execute(), feeds results back to the LLM, and emits the final run record.

AgentRuntime.invoke_iterative() exposes the same loop with streaming events over Server-Sent Events. run_stream() in ToolLoop emits token, tool_call, tool_result, and completed events suitable for the operator console.

Capability taxonomy

Agents declare capabilities drawn from a 25-entry taxonomy across five categories: Planning, Intake, Validation, Research, eXecution. The orchestrator and director use capabilities to route work and reason about deficits.

06Workflow engine

Workflows are declarative DAGs in YAML or JSON. The executor in engine/src/agent33/workflows/ performs a topological sort, runs steps with retries and timeouts, and persists checkpoints to PostgreSQL.

State machine

Each workflow run progresses through a deterministic state machine. Checkpoints record the last successful step output, so resuming from a SIGTERM mid-run is a primary path, not an emergency one.

07Memory and retrieval

AGENT-33 ships a four-stage retrieval stack:

  1. Short-term buffer. Last-N agent interactions, in-process, per session.
  2. Long-term store. pgvector-backed embeddings, 1536-dim, IVF-Flat cosine index, tenant-scoped.
  3. BM25 index. Lexical companion to dense retrieval, warmed up at startup against the long-term store.
  4. Hybrid searcher. Reciprocal rank fusion over BM25 + vector results.

Chunking

The ingestion pipeline uses a token-aware chunker (default 1200 tokens) so that embeddings are computed against units the LLM can re-emit verbatim. The chunker is pluggable; the default uses tiktoken-style heuristics with paragraph and sentence fallbacks.

Progressive recall

ProgressiveRecall resolves the recall budget per agent and per turn, starting with high-confidence summaries and expanding to full chunks only when the caller signals a deeper need. It composes with the skill injector so that prompts stay within the context budget.

Knowledge ingestion

The knowledge/ subsystem accepts RSS, GitHub, web, and folder sources via APScheduler-driven cron jobs. Ingestion respects per-tenant scopes; the schedule ownership guard prevents multiple replicas from running the same job.

08Tools and governance

Tools implement the Tool interface; schema-aware tools additionally implement SchemaAwareTool and ship a JSON Schema for their inputs. The registry validates the schema at registration and again on call via validated_execute().

Built-in tools

  • shell — command execution under CLIAdapter with timeout and allowlist
  • file_read — read a file under the workspace root
  • file_write — write a file under the workspace root
  • file_search — glob / ripgrep-style search
  • web_fetch — HTTP GET with SSRF protection and a host allowlist
  • browser — headless browser navigation and screenshot
  • code_exec — execute code via the sandbox contract

Governance

ToolGovernance consults the per-tenant allowlist, the active autonomy budget, and any pending approval tokens. The runtime answers allowed, denied, or needs-approval. Sensitive operations route through the HMAC-signed one-time approval token flow, with the operator console rendering pending requests.

09Deployment modes

The same engine image runs in three deployment shapes, each documented under docs/architecture/deployment-topologies.md.

M1

Lite

  • Single process
  • SQLite + in-process cache + in-process bus
  • Laptops, demos, CI smoke runs
  • Falls back here automatically when external stores are unreachable
M2

Standard, Compose

  • PostgreSQL + Redis + NATS + Ollama
  • Docker Compose
  • Default developer and operator profile
  • engine/docker-compose.yml
M3

Standard, K8s

  • Kustomize base manifests
  • StatefulSet Postgres + pgvector
  • Single-instance HPA by default
  • deploy/k8s/base/
M4

Enterprise, K8s

  • Production overlay
  • Ingress + TLS + persistent volumes
  • Dedicated observability stack
  • deploy/k8s/overlays/production/
M5

Air-gapped

  • Ollama-only path
  • Opt-in egress, no required cloud APIs
  • Pre-pulled images and models
  • Local pack registry
M6

Local-dev hybrid

  • Infra services in Compose
  • Engine on the host via uvicorn
  • Fast iteration loop
  • Used by the maintainer team

Authentication

Bearer JWT or X-API-Key on every non-public route. The auth middleware attaches a TokenPayload to request.state.user; routes consume it via FastAPI Depends. Scope enforcement is per-route, drawn from the scope catalogue in security/permissions.py.

Failure taxonomy

Every trace is tagged with one of ten failure categories:

  • F-ENV — environment / dependency unavailable
  • F-INP — bad input (validation, schema)
  • F-EXE — execution error
  • F-TMO — timeout
  • F-RES — resource exhausted
  • F-SEC — security policy denial
  • F-DEP — downstream dependency error
  • F-VAL — output validation failure
  • F-REV — review rejection
  • F-UNK — uncategorised

Retention policies are configured per category so operators can keep critical traces longer than noise.
