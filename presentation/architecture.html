1. System Overview

AGENT-33 is a FastAPI-based multi-agent runtime with a governance layer, a DAG-based workflow engine, hybrid retrieval-augmented memory, and a session-spanning evidence pipeline. The platform is structured as two layers: core/ (declarative artifacts — framework specifications, orchestrator rules, workflow templates) and engine/ (the running software: FastAPI app, executor, registries, persistence).

The split matters because governance enforcement happens between the runtime and the underlying I/O: a tool call is governed before the adapter executes, an autonomy budget is checked before a file is touched, and a high-risk action waits for review signoff before it proceeds.

2. Subsystem Map

The engine package (engine/src/agent33/) is organised into cohesive subsystems. Each subsystem owns its models, services, and (where applicable) API routes. Lifespan wiring is centralised in main.py.

3. Lifespan Initialisation Order

Subsystem initialisation in main.py follows a strict order. Each subsystem is stored on app.state; shutdown reverses the order. The order matters: registries need stores; the agent-workflow bridge needs registries; the LLM router needs the provider catalog.

Startup Order (dependencies left to right)
flowchart LR
  PG[PostgreSQL] --> RD[Redis] --> NA[NATS]
  NA --> AR[AgentRegistry]
  AR --> CE[CodeExecutor]
  CE --> LR[ModelRouter]
  LR --> EP[EmbeddingProvider + Cache]
  EP --> BM[BM25Index]
  BM --> HS[HybridSearcher]
  HS --> RP[RAGPipeline]
  RP --> PR[ProgressiveRecall]
  PR --> SR[SkillRegistry + Injector]
  SR --> AW[Agent-Workflow Bridge]
  AW --> ME[Memory: Observation + Summariser]
  ME --> TR[Training: optional]

Optional subsystems (training, AirLLM, advanced memory tiers) can be disabled via config without affecting the core runtime. The lifespan logs each subsystem on the way up and on the way down, so a startup failure tells you exactly which dependency broke.

4. Request Lifecycle

A single agent invocation passes through authentication, retrieval, prompt construction, the tool-use loop, and the trace pipeline. Every step is observed; nothing flows past a governance check by accident.

Invocation Sequence
sequenceDiagram
  participant C as Client
  participant A as Auth Middleware
  participant R as Agent Runtime
  participant I as Skill Injector
  participant M as Memory + RAG
  participant T as Tool Loop
  participant G as Tool Governance
  participant L as LLM Router
  participant O as Trace + Outcomes

  C->>A: POST /agents/{id}/invoke
  A->>A: Verify API key / JWT
  A->>A: Resolve tenant_id, scopes
  A->>R: Authorised invocation
  R->>I: Resolve skills (L0/L1/L2)
  R->>M: Hybrid retrieval (BM25 + pgvector + RRF)
  R->>L: Render prompt, stream completion
  L-->>R: Completion or tool call
  R->>G: Check tool allowlist for agent role
  G->>T: Execute tool (governed)
  T-->>R: Tool result
  R->>L: Continue tool-use loop
  L-->>R: Final answer
  R->>O: Emit trace + outcome
  R-->>C: Response + trace_id

The iterative path (AgentRuntime.invoke_iterative()) resolves effort-routing parameters after pre-hooks run, so hook mutations to the input affect routing and model selection.

5. Agent Runtime

The agent runtime is the heart of the platform. It constructs the prompt from a JSON definition, injects skills via L0/L1/L2 progressive disclosure, runs the tool-use loop, and streams results back to the caller.

Definition (declarative)

  • System prompt + role description
  • Capability tags (P / I / V / R / X)
  • Allowed tools + governance scope
  • Default LLM + effort routing config

Runtime behaviours

  • Hybrid memory retrieval before prompt assembly
  • Skill injection at the right tier
  • Tool loop with iteration budget
  • Trace emission on every step

The runtime supports a streaming path (run_stream()) that preserves the final output on max_iterations exits and still propagates real TypeErrors from the underlying stream call instead of swallowing them as transient retries.

6. Workflow Engine

Workflows are YAML DAGs. The executor topologically sorts the graph, runs steps with retries and timeouts, fans out via parallel groups, branches on conditionals, and persists checkpoints so long-running workflows survive restarts.

The workflow bridge (invoke_agent.py) has set_definition_registry() as a fallback path so workflow execution remains decoupled from the FastAPI app state when invoked programmatically (e.g., scheduled runs).

7. Memory & Retrieval

The memory stack has four tiers: a short-term buffer in Redis, a long-term store backed by PostgreSQL + pgvector, a BM25 lexical index for keyword match, and a hybrid searcher that combines BM25 and vector scores via reciprocal-rank fusion.

Retrieval Pipeline
flowchart LR
  Q[Query] --> ST[Short-term buffer]
  Q --> BM[BM25 Index]
  Q --> EMB[Embedding Provider + Cache]
  EMB --> PV[pgvector Search]
  BM --> RRF[Reciprocal Rank Fusion]
  PV --> RRF
  RRF --> PR[Progressive Recall]
  PR --> CTX[Context Pack]
  ST --> CTX

Token-aware chunking (1,200 tokens per chunk) keeps embeddings within provider limits and makes retrieved spans coherent. The BM25 index can warm up at startup; tests that mock LongTermMemory.initialize() and close() must also mock scan() or disable BM25 warmup, otherwise lifespan startup will trip on the warmup path.

8. Governance Contracts

Governance is enforced at four layers: tool allowlists, autonomy budgets, review signoff, and release lifecycle gating. None of these are advisory; the runtime refuses to proceed past a failed check.

Tool governance This agent role may invoke this tool Refuse with F-SEC failure
Autonomy budget File / command / network scope per invocation Block or escalate to review
Review automation Human signoff on high-risk actions Hold until decision; record F-REV

Trace pipeline

Every invocation produces a trace record with the failure category (10 taxonomy entries), per-step timings, tool calls, prompts, and completions. Retention policies are configured per tenant. Traces are replayable: given the trace_id, the platform can reconstruct the exact invocation.

Outcomes & evaluation

The outcomes store records every invocation result for downstream evaluation. Golden tasks + regression detection compare current behaviour to historical baseline. Evaluation gates can block deployment if regression exceeds threshold.

12. Deployment Topologies

Three deployment shapes share the same image and the same code paths. The difference is what supporting infrastructure runs alongside, and which optional subsystems are enabled.

Lite

Single process + Postgres + Ollama. Skips Redis cluster, NATS, and review automation.

Best for: laptops, dev, one-team pilots.

Standard

Full stack via Docker Compose: Postgres + pgvector, Redis, NATS, Ollama. Multi-tenant. Trace pipeline on.

Best for: small-to-mid teams; production for low-mid volume.

Enterprise

Kubernetes deployment with HPA, durable stores, review automation, autonomy budgets, full observability.

Best for: production, multi-tenant SaaS, regulated environments.
