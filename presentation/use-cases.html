Where AGENT-33 Fits

Seven scenarios where a governed multi-agent platform matters more than a slick single-shot chat. Each one tells you the topology to deploy, the subsystems to flip on, and the operational shape to expect.

7 scenarios v2.1.0 Apache 2.0

GOV Governed Internal Agent Platform Scenario 01

Your company has half a dozen would-be "AI projects" each spinning up their own LangChain stack, prompt store, and shadow OpenAI key. You need a single platform that lets teams ship agents safely without handing every team its own production runtime to operate.

The problem

Without a common runtime, every team reimplements tool governance, retries, observability, and tenant isolation. Each team picks its own LLM provider. There’s no single place to see who is calling what, no consistent audit trail, and no way to enforce an organisational policy on which tools an agent is allowed to invoke.

AGENT-33’s answer

  • One FastAPI runtime, one tenant model, one auth layer. Teams ship JSON agent definitions, YAML workflows, and Markdown skills — not their own infrastructure.
  • Tool registry + governance allowlists let platform engineering define which tools each agent role may invoke; rules are enforced at execution time.
  • Trace pipeline + outcomes store give security and ops a single dashboard for every invocation, not seven dashboards in seven repos.
  • Provider-agnostic LLM router lets each team pick its preferred model (Ollama, OpenAI, Anthropic, llama.cpp) without forking the platform.

Operational outcome

Platform team operates one stack. Application teams ship agents through declarative artifacts and PR reviews. Compliance has one trace store to subpoena. Cost reporting rolls up cleanly by tenant.

RAG RAG-with-Audit for Regulated Knowledge Scenario 02

A regulated business wants Retrieval-Augmented Generation across internal policy and procedure documents. Every answer must cite specific sources and the whole pipeline must be reproducible during audit.

The problem

Pure vector-search RAG produces fluent answers with no way to say which chunk drove which sentence. SaaS RAG products often refuse to run on-prem, refuse to expose their embedding model, or refuse to let you pin a chunking strategy. Audit teams want to verify that a given answer would still be produced from the same evidence today.

AGENT-33’s answer

  • Hybrid retrieval (BM25 + pgvector with reciprocal-rank fusion) gives both lexical and semantic recall; no SaaS dependency for the index.
  • Token-aware chunking (1,200 tokens) is deterministic and reproducible; the chunk fingerprint is in the trace.
  • Lineage and replay systems record which documents and chunks fed each invocation; the same prompt + index snapshot = the same answer (modulo model nondeterminism).
  • Embedding provider is pluggable; you can run sentence-transformers locally or call a provider, your call.

Operational outcome

The knowledge corpus lives in PostgreSQL on infrastructure you control. Every response carries its retrieval context in the trace store. Audit can replay any invocation and verify that the produced answer matched what evidence supported.

SaaS Multi-Tenant Agent SaaS Scenario 03

You’re building an agent-driven SaaS where each customer brings their own data, their own LLM key, and their own configuration. The platform must keep tenants isolated end-to-end with no chance of crosstalk in workflow state, memory, traces, or output artifacts.

The problem

Most agent libraries treat multi-tenancy as a feature to add later. The result: every model, every queue, every cache is single-tenant by default, and bolting on tenant_id retroactively is the foundation for a future security incident. Customers can demand per-tenant encryption keys, per-tenant rate limits, and per-tenant trace export.

AGENT-33’s answer

  • Multi-tenancy is in the model layer: every persisted row carries tenant_id; auth middleware resolves tenant from API key or JWT.
  • Per-tenant encryption keys, scopes, rate limits, and webhook configurations are first-class.
  • Workflow state, memory store, lineage records, traces, and outcomes are all scoped by tenant.
  • Per-tenant cost tracking (tokens, invocations) feeds the billing pipeline cleanly.

Operational outcome

Provision a new customer by adding a tenant row plus their LLM provider credentials in the vault. Every subsequent invocation is correctly scoped without per-customer code changes. Compliance reports filter by tenant trivially.

RES Research & Investigation Workflows Scenario 04

A research team needs to run repeatable multi-step investigations: browse, scrape, summarise, cross-reference, and produce a structured report. Today they string together notebooks and one-off scripts. They want a workflow engine with retries, parallelism, and provenance.

The problem

Notebook-driven investigations are brittle. Cells go stale, environments drift, and rerunning a six-month-old query produces different output for reasons no one remembers. Cloud agent products do not always expose the workflow graph as code, which makes investigations hard to version-control and hard to inherit.

AGENT-33’s answer

  • YAML workflow definitions are checked into git like any other artifact; the topological-sorted DAG executor runs them deterministically.
  • Parallel groups let the researcher fan out across sources; the engine joins, deduplicates, and assembles the report.
  • Checkpoint persistence means a long-running workflow survives a restart; the resume point is exact.
  • Browser agent + web-fetch tool give scripted browsing without writing Selenium boilerplate.
  • Evidence capture — every step’s input, output, and tool calls go into the trace store and lineage system.

Operational outcome

The team commits its workflows. Past investigations are reproducible. New analysts inherit the methodology by reading a YAML file, not by pair-programming with senior researchers for two weeks.

LCL Local-First Workstation Agent Scenario 05

A developer or analyst wants to run a serious agent platform on a single laptop or workstation — including the model — without rebuilding the stack to do it. No cloud account, no per-token billing, no outbound data plane during normal operation.

The problem

Most agent platforms assume the cloud as default and treat local as an afterthought. The trial-by-cloud pattern means experimenting with a new agent design costs real money each iteration, and any sensitive data has to be triple-checked before it leaves the box.

AGENT-33’s answer

  • Ollama is the default LLM provider; pull a model, point AGENT-33 at it, you’re live.
  • Lite runtime mode skips the heavy subsystems (no Redis cluster, no NATS, no review automation) and runs out of one process.
  • Local pgvector + SQLite checkpoints give real persistence without an ops burden.
  • Switch on remote providers (OpenAI, Anthropic, llama.cpp) per agent, per workflow, or per invocation when you do want them.

Operational outcome

A single command brings up the stack; a single command tears it down. You can iterate on agent design at zero marginal cost. Sensitive data never leaves the box unless you explicitly route a workflow through a remote provider.

MIX Multi-Provider Model-Mix Exploration Scenario 06

An ML team wants to compare provider behaviour — same workflow, same data, different models — without rewriting the workflow for each. They also want to route the cheap reasoning steps to a small local model and reserve the big remote model for the synthesis step.

The problem

Hardcoding a single LLM provider in workflow code couples the investigation to one vendor and one price point. Effort-based routing (small model for triage, big model for synthesis) is the obvious optimisation but most agent libraries leave you to wire it by hand.

AGENT-33’s answer

  • 20+ LLM providers auto-register from environment variables; the model router selects per agent definition.
  • Effort-based parameter resolution chooses the right model for the right step (cheap reasoning vs heavy synthesis).
  • The workflow stays the same; the provider strategy swaps via configuration.
  • Outcomes store + evaluation gates let you compare side-by-side on real workloads, not just on a benchmark.

Operational outcome

The team runs A/B comparisons across providers without code changes. Routing policy is a configuration artifact. Cost tracking by tenant and by step gives a clear picture of which provider is paying for itself.

REG Regulated / On-Prem Environments Scenario 07

A regulated business (finance, healthcare, public sector) needs an agent platform that runs entirely on infrastructure they control, with no mandatory outbound network calls and a credible audit story. They have Kubernetes, they have Prometheus, they have a CISO with opinions.

The problem

Cloud-only agent platforms are off-table by procurement. Building from scratch is multi-quarter and high-risk. Existing open-source orchestration libraries treat governance as someone else’s problem. What’s wanted is a stack that boots inside the existing security envelope and produces evidence the audit team can act on.

AGENT-33’s answer

  • Apache 2.0, self-hosted, no mandatory outbound calls; Ollama or llama.cpp keeps inference local.
  • Per-tenant encryption keys, scopes, and rate limits; auth via API key or JWT.
  • Trace pipeline + outcomes + lineage + replay give the auditor a defensible answer to "what happened, why, and would it happen again."
  • Autonomy budgets cap what an agent may do per invocation (file scope, command scope, network scope) with explicit escalation.
  • Review automation gates can require human signoff on high-risk actions before they execute.

Operational outcome

The platform sits in the existing Kubernetes footprint behind the existing network perimeter. Metrics flow to the existing Prometheus. Traces flow to the existing OpenTelemetry collector. The audit team can subpoena one store and get one answer.

Where AGENT-33 is not a fit

Honest scoping. Pick a different tool for these.

A single-shot chat experience for end users

  • If you want a polished consumer chat interface with built-in voice, image generation, and a customer billing portal, a hosted chat product will be faster to deliver than configuring AGENT-33 yourself.
  • AGENT-33 is a platform for building governed agent workloads. It is not a consumer chat UI.

A vector database or a feature store

  • AGENT-33 uses pgvector for retrieval but it is not a managed vector DB; it is not a substitute for dedicated retrieval infrastructure if you already have one.
  • If your team already runs a feature store or vector DB, AGENT-33 can call into it via tools and skills instead of replacing it.

A generic web application framework

  • FastAPI lives inside AGENT-33, but the platform is opinionated about what it does (run agents, run workflows, govern tools, persist evidence). Bolting a generic web app onto it works but is not the design target.
  • Use FastAPI directly if the goal is a CRUD service that happens to call an LLM occasionally.

A model training / fine-tuning platform

  • AGENT-33 hosts an optional training subsystem hook, but it is not an MLOps platform; if your workload is GPU-bound model training, the right tools are PyTorch Lightning, Hugging Face TRL, or a managed training service.
  • AGENT-33 is for orchestrating workloads on top of already-trained models.
