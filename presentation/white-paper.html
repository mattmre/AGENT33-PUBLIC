Abstract

AGENT-33 is an open-source multi-agent orchestration framework engineered for teams who need real workflows, explicit governance, and a usable control plane instead of a pile of disconnected scripts. It combines a declarative agent registry, a DAG-based workflow engine, a sandbox-contract code executor, a hybrid BM25 + vector retrieval stack, and a trace pipeline with a 10-category failure taxonomy in one async Python runtime.

The platform is multi-tenant from day one, local-first by default (Ollama out of the box), and exposes six declarative extension points: agents, workflows, skills, packs, tools, and MCP servers. Twelve subsystems carry governance, evidence, and lifecycle responsibilities so that an operator can answer “who did what, when, and under what authority” without leaving the platform.

This document describes the design philosophy, system architecture, and operational characteristics that distinguish AGENT-33 from generic LLM orchestration libraries and hosted agent platforms.

1Motivation

1.1 The agent-platform problem

The 2020s produced an abundance of LLM endpoints and a smaller number of “agent frameworks”: libraries that wrap a tool-use loop, a prompt template engine, and a vector store. Most of them ship as developer libraries and assume the application layer above them will solve the platform concerns.

That assumption holds for a single-user agent script. It does not hold for:

  • Multi-tenant SaaS deployments where one operator’s data must never leak into another’s context
  • Regulated workloads where every tool call needs an identity, a scope, and an audit entry
  • Teams running agent rollouts where evaluation gates and regression detection are non-negotiable
  • Self-hosted environments where cloud APIs are off the table

Each of these introduces requirements that are easier to satisfy at the platform layer than to retrofit per application. AGENT-33 is the platform layer.

1.2 The governance problem

An agent is, by construction, a system that can invoke tools and write outputs on a human’s behalf. The accountability gap is wider than for traditional software: the model decides which tool to call, with what arguments, in what order. Without explicit governance, the application owner cannot answer the post-hoc question of why a particular tool was invoked, by which agent, against which tenant, with what authority.

AGENT-33 closes that gap in three ways:

  1. Identity per call. The auth middleware resolves a tenant and a token payload for every request. Tool calls carry that identity through to execution and persistence.
  2. Scope per route. Tools are JSON-Schema validated. Tool execution is allowlist-gated per tenant. Autonomy budgets enforce file, command, and network scope at runtime.
  3. Evidence per action. Tool invocations produce traces. Workflow steps produce checkpoints. Agent runs produce lineage records. Replays survive a fresh checkout.

1.3 The local-first imperative

A significant fraction of agent workloads cannot send data to cloud APIs. The reasons vary — regulatory regime, contractual obligation, data residency, operator preference — but the engineering consequence is the same: the platform must run end-to-end against a model the operator controls.

AGENT-33’s default stack is Ollama. The Compose bootstrap brings the entire runtime up against a local Ollama endpoint with no external dependencies. Cloud providers, hosted vector databases, signing authorities, and message buses are all opt-in. The model router auto-registers 20+ providers from environment variables, so swapping inference backends is configuration, not code.

2Design Principles

2.1 Declarative-first

Agents are JSON. Workflows are YAML or JSON. Skills are Markdown with YAML frontmatter. Packs are YAML manifests over a directory of files. Anything you can declare you can ship without code changes. The auto-discovery scan at startup picks up files dropped into the appropriate directory.

Declarative-first does not mean “declarative-only.” The Tool interface, the messaging adapter interface, the model provider interface, and the sandbox adapter interface are all extensible in Python. Declarative is the default; programmatic extension is the escape hatch.

2.2 Governed execution

Code execution, tool calls, autonomy budgets, review sign-offs, release lifecycles, and pack trust all flow through dedicated subsystems with explicit state machines. There is no implicit “agent does whatever the model said” path through the runtime. The default answer to a privileged operation is no; the operator’s job is to widen scope explicitly, not to narrow it after-the-fact.

2.3 Local-first, remote optional

The default stack runs against Ollama on a developer laptop. Every external integration is opt-in behind feature flags and graceful fallbacks. The lifespan treats Redis and NATS as optional; failed initialisation logs a warning and falls back to InProcessCache and InProcessMessageBus.

The application starts in lite mode if PostgreSQL, Redis, and NATS are all unavailable. This is intentional: the platform must be self-sufficient on a laptop or in a CI smoke run, not just in production.

2.4 Multi-tenant by construction

Tenant id is a property of the durable schema, not a deployment convention. Every Alembic-managed model carries a tenant_id column. The auth middleware resolves the tenant from the inbound credential and attaches it to the request state. Repository methods either filter by tenant_id directly or call a tenant-scoped helper that does.

Cross-tenant data is a bug, not a misconfiguration. Operator and admin scopes still see one tenant at a time unless they explicitly use admin endpoints. The test suite includes regression cases that exercise tenant isolation at the repository layer.

2.5 Evidence over assertion

The platform proves what happened, rather than asserting it. Traces, lineage, and replay records survive long after the agent loop exits. The trace pipeline tags every span with one of ten failure categories so that operators can group incidents by class and apply retention policies that match their compliance posture.

3Architecture

3.1 Two-layer model

AGENT-33 separates specification from runtime. The two layers ship together and version together but serve different purposes.

  • core/ — Plain Markdown specifications: agent role specs, orchestrator handoff and evidence rules, phase planning and gates, versioned API specs, workflow templates, policy pack definitions. These documents are auto-loaded by the orchestrator agent at runtime.
  • engine/ — Async Python runtime. Hatchling-built, strict mypy, Ruff linting, pytest in async mode. This is the actual software.
  • frontend/ — React / TypeScript single-page application. The day-to-day operator interface; every operation is reachable via the API and CLI.

3.2 Subsystem map

The engine package contains roughly 60 first-class subsystem directories under engine/src/agent33/. They cluster into seven groups by responsibility: product surfaces, platform, integration, observability, control surfaces, runtime substrate, and foundation. The architecture deck and component catalog walk through each directory and the modules to read first.

3.3 Lifespan and request lifecycle

The FastAPI lifespan brings subsystems up in a deterministic order, storing each on app.state.*. Shutdown is the reverse: workflows are cancelled, services drain, message buses disconnect, and the database connection pool closes last. The technical brief documents the full sequence; the architecture deck has the sequence diagram.

4Agent Model

4.1 Declarative definitions

Agents are JSON files auto-discovered from engine/agent-definitions/. Each definition declares the agent’s capabilities (drawn from a 25-entry taxonomy across P/I/V/R/X categories), the tools it may use, the LLM model and parameters, the system-prompt frame, and the skills it loads.

Six reference agents ship in the box: orchestrator, director, worker, qa, researcher, and browser-agent. They are not the only agents possible — they are the curated starting set that the operator console and the workflow library reference by name.

4.2 Invocation paths

AgentRuntime exposes three invocation paths:

  1. invoke() — synchronous, returns the final AgentRunResult.
  2. invoke_iterative() — streams tool-use events over SSE; suited for the operator console.
  3. run_stream() via ToolLoop — the underlying generator that emits token, tool_call, tool_result, and completed events.

Iterative routing parameters (model selection, temperature, max iterations) are resolved after pre-hooks run, so hook-mutated inputs influence routing instead of stale request payloads.

4.3 Tool-use loop

The tool-use loop is the heart of the agent runtime. A single iteration looks like:

  1. The runtime opens a trace, runs progressive recall against memory, and constructs the system prompt with skills injected per the L0/L1/L2 disclosure ladder.
  2. The LLM is called with the allowed tools and the assembled context.
  3. For each tool call the model emits, the runtime asks ToolGovernance.authorize() for the tool, parameters, and current autonomy budget.
  4. If authorised, the registry runs validated_execute(), which JSON-Schema validates the inputs again and executes the tool.
  5. The result is fed back to the LLM as a tool-result message. The loop continues until the model emits a final message or the iteration ceiling is reached.

4.4 Skill injection

Skills are SKILL.md or SKILL.yaml files with YAML frontmatter. The skill registry scans the skill directories on startup and on demand; the skill injector chooses a disclosure level per call:

  • L0 — one-sentence summary; cheap, used for many skills at once
  • L1 — outline; medium, used when L0 keeps surfacing the same skill
  • L2 — full body; expensive, used when the agent has committed to the skill

Tool context is resolved at injection time so that L2 skills can reference the tools the agent currently has scoped, not the tools the skill was authored against.

5Workflow Execution

5.1 DAG executor

Workflows are declarative DAGs in YAML or JSON. The executor performs a topological sort, runs steps in dependency order, and persists checkpoints to PostgreSQL. Steps with no inter-dependencies run in parallel; parallel groups are a first-class action.

5.2 Action library

Eight action types cover the common shapes: invoke_agent, run_command, validate, transform, conditional, parallel_group, wait, and execute_code. Each action takes inputs from earlier steps via an expression evaluator and writes outputs that downstream steps can reference.

5.3 Checkpoints and resume

Each successful step writes a checkpoint to PostgreSQL with the step output, the run state, and the expression context. A workflow that hits a SIGTERM mid-run resumes from the last checkpoint; the topological sort recomputes the remaining DAG against the checkpoint state. Resume is a primary path, not an emergency one.

5.4 Retries and timeouts

Each step has a retry policy (count, backoff) and a timeout. The executor wraps step execution with the policy; failed steps are retried with exponential backoff until exhaustion. Timeout enforcement is cooperative: long-running tool calls respect cancellation tokens.

6Memory + Retrieval

6.1 Four-stage stack

The retrieval stack has four stages:

  1. Short-term buffer — last-N agent interactions, in-process, per session.
  2. Long-term store — pgvector-backed embeddings, 1536-dim, IVF-Flat cosine index, tenant-scoped.
  3. BM25 index — lexical companion to dense retrieval, warmed up at startup.
  4. Hybrid searcher — reciprocal rank fusion (RRF) over BM25 + vector.

6.2 Chunking

The ingestion pipeline uses a token-aware chunker (default 1200 tokens) so that embeddings are computed against units the LLM can re-emit verbatim. Chunk size is configurable per ingestion source; the chunker preserves paragraph and sentence boundaries where possible.

6.3 Hybrid retrieval

Vector retrieval is good at semantic similarity; BM25 is good at lexical match. AGENT-33 runs both for every query and fuses the rankings with reciprocal rank fusion (RRF). The default k for RRF is configurable per tenant; the default result set size respects the agent’s recall budget.

6.4 Progressive recall

ProgressiveRecall resolves the recall budget per agent and per turn, starting with high-confidence summaries and expanding to full chunks only when the caller signals a deeper need. It composes with the skill injector so that prompts stay within the context budget.

7Governance Contracts

7.1 Tool governance

Every tool call passes through ToolGovernance before execution. The check considers:

  • The per-tenant allowlist for the tool
  • The active autonomy budget for the subject (file scope, command scope, network scope)
  • Pending approval tokens for sensitive operations
  • Per-subject rate limits

The answer is allowed, denied, or needs-approval. Sensitive operations route through the HMAC-signed one-time approval token flow; the operator console renders pending requests and forwards approvals back to the runtime.

7.2 Autonomy budgets

An autonomy budget is an explicit declaration of what an agent or workflow run may do. It has a lifecycle state machine (DRAFT → ACTIVE → COMPLETED), preflight checks (PF-01..PF-10), and runtime enforcement (EF-01..EF-08) covering file access scope, command scope, and network egress scope.

Stop conditions and escalation policies are configured per budget. A budget that hits a stop condition triggers a structured escalation: the run pauses, the operator is notified, and the trace pipeline records the cause. Resumption requires an explicit operator action.

7.3 Review automation

The review/ subsystem implements a two-layer review process: risk assessment runs against the change set or the run output, reviewer assignment picks the smallest reviewer set that covers the risk surface, and the signoff state machine drives the review to either approved or rejected. The review service exposes an HTTP API; the operator console renders the queue.

7.4 Release lifecycle

Releases follow a lifecycle state machine: PLANNED → FROZEN → RC → VALIDATING → RELEASED → ROLLED_BACK. A pre-release checklist (RL-01..RL-08) gates the transition from FROZEN to RC. The sync engine runs in dry-run mode by default and uses fnmatch matching to scope sync operations. The rollback manager has a decision matrix that drives the response to validation failures.

8Observability

8.1 Trace pipeline

Every run produces a structured trace following the Session → Run → Task → Step → Action hierarchy. The trace collector tags each trace with a failure category drawn from a 10-entry taxonomy: F-ENV, F-INP, F-EXE, F-TMO, F-RES, F-SEC, F-DEP, F-VAL, F-REV, F-UNK. Retention policies are configured per category so that operators can keep critical traces longer than noise.

8.2 Lineage and replay

ExecutionLineage records the parent-child graph of agent runs, workflow runs, and sub-agent dispatches. ExecutionReplay records enough state to deterministically replay a recorded run on a fresh checkout. Lineage answers “what produced this?”; replay answers “produce it again.”

8.3 Evaluation gates

The evaluation/ subsystem exposes golden tasks (GT-01..GT-07), golden cases (GC-01..GC-04), a metrics calculator (M-01..M-05), a gate enforcer with configurable thresholds, and a regression detector (RI-01..RI-05). The evaluation service is called by the release lifecycle: the transition from VALIDATING to RELEASED is gated on a passing evaluation run.

8.4 Outcomes and improvement

The outcomes/ subsystem records the operational outcome of each agent run and workflow run, persisted to SQLite (best-effort) for impact tracking. The improvement/ subsystem aggregates outcomes into research intake records, lessons learned with action tracking, and improvement metrics (IM-01..IM-05) with trend computation. The roadmap refresh loop runs on a cadence.

9Security Model

9.1 Authentication and authorisation

Bearer JWT or X-API-Key on every non-public route. The auth middleware attaches a TokenPayload to request.state.user; routes consume it via FastAPI Depends. Public paths are an explicit list (/health, /healthz, /readyz, /metrics, /v1/auth/token, dashboard reads, OpenAPI docs); everything else requires a credential.

9.2 Scopes

The scope catalogue covers admin, agents:read, agents:write, agents:invoke, workflows:read, workflows:write, workflows:execute, tools:execute, operator:read, and operator:write. Routes declare the scopes they require; the middleware enforces them.

9.3 Encryption and secrets

AES-256-GCM encryption for secrets at rest. Fernet-backed vault for high-level secret storage. The configuration system uses SecretStr for sensitive fields; callers must call get_secret_value() explicitly. The lifespan validates production-grade secrets at startup and refuses to start with the development default JWT secret in production mode.

9.4 Pack trust

Pack integrity is verified with SHA-256 via hmac.compare_digest. The pack trust manager applies a per-pack trust policy and consults a revocation list before activation. Asymmetric authorship signing via Sigstore cosign is on the near-term roadmap.

9.5 Sandboxing

Code execution runs under CodeExecutor with a SandboxConfig, an ExecutionContract, and an ExecutionResult. Three adapters ship: CLIAdapter for subprocess execution with timeout, JupyterAdapter for kernel-based execution, and an optional GPU Docker adapter for accelerator workloads.

10What AGENT-33 Is Not

Honest scoping matters as much as honest capability claims. The following are explicitly outside the platform's responsibility.

10.1 Not a model provider

AGENT-33 does not train, fine-tune, or serve LLMs in production. It is a client of model endpoints. The model router auto-registers providers but does not run inference itself. AirLLM-backed local inference is an optional Compose profile, not a core path.

10.2 Not a vector database

AGENT-33 reads and writes embeddings into pgvector. It does not replace purpose-built vector databases at high scale. Operators who need billion-vector-class retrieval should plug a dedicated vector store into the embedding provider interface; the hybrid searcher will compose over it.

10.3 Not a generic web framework

AGENT-33 is built on FastAPI, but it is not a general-purpose web application scaffold. The routes, middleware, and lifespan are tuned for agent workloads. Operators who want to add unrelated CRUD endpoints should run a separate FastAPI app behind the same gateway.

10.4 Not a replacement for purpose-built MLOps

The evaluation subsystem covers agent-output regression detection. It is not a replacement for full MLOps experiment tracking. Operators running deep fine-tuning experiments should integrate with their MLOps stack via the evaluation API.

11Roadmap

The near-term public direction:

  • Ecosystem growth — broader pack catalog, community-contributed skills and tools, signed pack distribution via Sigstore cosign.
  • MCP integrations — richer hosted MCP server surface and tighter MCP client interop with the agent runtime.
  • Public benchmarking — continued evaluation against SkillsBench with CTRF reporting and weekly full-tier runs.
  • Provider depth — first-class support for additional local-inference backends (llama.cpp, LM Studio, AirLLM) and embedding providers.
  • Operator UX — visual workflow builder polish, sub-agent execution trees, knowledge ingestion cron expansion.
  • Horizontal scaling — multi-replica deployment with explicit state-boundary coordination, building on the existing ownership guard.
Known limitations at launch: The default Kubernetes manifest is single-instance. Pack signing uses SHA-256 integrity verification only (asymmetric authorship signing is upcoming). A handful of authenticated routes are not yet scoped per-tenant. These are tracked, not hidden.
